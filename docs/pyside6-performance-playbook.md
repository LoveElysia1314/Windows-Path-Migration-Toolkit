# PySide6 性能优化可迁移笔记（实战版）

本笔记提炼自多轮真实优化实践，目标是形成可复用的方法论，而不是绑定某个项目细节。

## 1. 先判断：慢在“计算”还是“渲染”

很多 GUI 卡顿不是业务算法慢，而是主线程渲染阻塞。

建议把一次操作拆成至少 3 个指标：
- 过滤/计算耗时
- 回到主线程的切换开销
- UI 渲染/布局耗时

实践经验：
- 当“结果集突然变大”时，主线程渲染通常是主瓶颈。
- 单纯给过滤做缓存，往往只能改善一部分，不会根治卡顿。

## 2. 最有效的结构升级：QTableWidget -> QTableView + QAbstractTableModel

### 为什么有效
- `QTableWidget` 是“项对象”模式：每个单元格一个对象，行数大时对象创建和布局成本高。
- `QTableView + QAbstractTableModel` 是“模型视图”模式：按需取数，避免大量无谓对象创建。

### 可迁移结论
- 只要列表规模可能到几百/几千行，优先用 Model/View。
- 性能收益通常来自“减少对象生命周期管理”，而非某一条小技巧。

## 3. 过滤链路的优雅落点：QSortFilterProxyModel

### 统一过滤职责
把这些条件统一放到代理层：
- 搜索词
- 路径排除（如系统目录、用户目录）
- 关键词黑名单

### 带来的价值
- 降低窗口类复杂度（少写一条长过滤流水线）
- 减少状态同步 bug（旧结果覆盖新条件）
- 新增过滤条件时改动集中（维护成本低）

### 迁移建议
- 源模型只负责“提供数据”。
- 代理模型只负责“筛选/排序规则”。
- 视图只负责“展示和交互”。

## 4. 对高频输入做“合适的防抖”而不是统一防抖

经验值（可按项目微调）：
- 复选框切换：短防抖（约 30-80ms）
- 搜索输入：长防抖（约 250-400ms）

原因：
- 复选框是离散操作，用户预期快反馈。
- 搜索是连续输入，过快触发会引发重复过滤与重绘。

## 5. 滚动/拖动进度条卡顿的常见真因：渲染时触发重 I/O

### 典型表现
- 拖动滚动条到新位置，瞬间卡一下。
- 不是“未渲染”本身的问题，而是“临时渲染时顺带做了重操作”（常见是图标加载）。

### 有效方案（组合拳）
1. 图标占位
- 首帧先显示默认图标，避免 paint 路径中阻塞 I/O。

2. 分批懒加载
- 用 `QTimer` 分批加载真实图标（例如每批 8~20 个）。

3. 局部刷新
- 图标加载完成后，仅 `dataChanged` 相关行/列，不做全表刷新。

4. 滚动预热
- 在可视区上下预加载一段范围，降低“跳转后第一屏”抖动。

## 6. 预计算字段：把重复字符串处理前移到扫描阶段

把以下内容预先计算并缓存到每行数据：
- `_search_blob`：用于搜索匹配的大写/小写归一化文本
- `_keyword_blob`：用于关键词黑名单匹配
- `_is_standard_path` / `_is_users_path`：布尔标记
- `_icon_key`：规范化后的图标路径键

收益：
- 过滤阶段减少重复拼接与规范化
- 代码更清晰，规则判断更直接

## 7. 缓存策略的正确使用姿势

### 何时有用
- 计算确实重，且相同条件会反复出现。

### 何时收益有限
- 主瓶颈在 UI 渲染时，缓存只能“治标不治本”。

### 正确性注意
- 数据源变化（重新扫描）后必须失效缓存。
- 缓存键要覆盖真实影响条件。

## 8. 视图层反模式（尽量避免）

1. 大表格使用 `ResizeToContents` 作为默认列策略
- 会触发大量测量，尤其在频繁更新时。
- 更推荐固定宽度 + 关键列 `Stretch/Interactive`。

2. 在循环内频繁触发布局/刷新
- 优先批处理后一次性通知。

3. 把磁盘访问放进 `data()` 的热路径且无缓冲
- 会把滚动直接变成 I/O 驱动。

## 9. 选择状态管理建议

- 勾选状态建议按“稳定 ID 集合”存储，而不是依赖行号。
- 排序/过滤后状态不易错乱。
- 批量全选/清空时优先“一次性更新集合 + 批量发信号”。

## 10. 一套可复用的优化流程

1. 先打点：测“过滤 vs 渲染”。
2. 若渲染占主导：先改架构（Model/View）。
3. 过滤迁移到 ProxyModel，降低控制器复杂度。
4. 图标/缩略图改懒加载 + 分批 + 局部刷新。
5. 再做小优化：列策略、防抖参数、预热范围。
6. 每一步都回归验证，避免“优化引入新抖动”。

## 11. 经验阈值（实践参考，不是硬规则）

- 行数 > 500：优先考虑 Model/View。
- 行数 > 1000 且有图标列：必须考虑懒加载和分批。
- 搜索输入：300ms 左右防抖通常比 100ms 更稳。
- 图标批次：8~20 通常较平衡（需按机器调参）。

## 12. 小结

真正可迁移的核心不是“某个参数”，而是三条原则：
- 把重活从主线程热路径移开。
- 把职责放到正确层次（Model / Proxy / View）。
- 先定量诊断，再针对性优化。

当你把这三条做到位，大多数 PySide6 大列表卡顿问题都能明显改善。
